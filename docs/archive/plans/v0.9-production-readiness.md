# v0.9 Production Readiness Plan

## Overview
Prepare glory-hole for production deployment by addressing performance bottlenecks, improving code maintainability through refactoring, removing technical debt, and increasing test coverage in critical areas.

**Goal**: Production-ready v0.9 with improved performance, better code organization, and comprehensive test coverage.

**Estimated Effort**: 5-7 developer days

---

## Phase 1: Performance & Critical Fixes (HIGH PRIORITY)

### 1.1 Benchmark Whitelist & Blocklist Lookup Throughput (TD-002)
**Context**: The whitelist already uses lock-free atomic maps, but we lack benchmark data that proves the new impl holds up under sustained query load and documents the win over the former RWMutex version.

**Objectives**:
- Capture baseline QPS/latency for whitelist + blocklist lookups under realistic traffic.
- Detect regressions when pattern matchers grow large (e.g., >10k entries).
- Document the findings and wire them into CI so future changes can be compared.

**Implementation**:
- Add focused benchmarks in `pkg/dns/server_test.go` (or a new `server_bench_test.go`) that exercise `shouldBlockDomain` / whitelist checks with varying list sizes.
- Create a load scenario in `test/load/dns_load_test.go` that hammers whitelisted domains vs blocked domains and records mean/percentile latency.
- Expose benchmark results via `make bench-whitelist` target (wrapper script).
- Update the engineering docs with the measurements and acceptable thresholds (e.g., 2x safety margin against v0.8 numbers).

**Files to Create/Modify**:
- Modify: `pkg/dns/server_test.go` (benchmarks + helpers)
- Modify: `test/load/dns_load_test.go` (new scenario + reporting)
- Modify: `Makefile` (bench target)
- Update: `docs/performance/README.md` (benchmark table)

**Testing**: `go test -run=Bench -bench=Whitelist ./pkg/dns` and `go test -run TestDNSLoadWhitelist ./test/load` with assertions on max latency / QPS.

**Estimated Time**: 6-8 hours
